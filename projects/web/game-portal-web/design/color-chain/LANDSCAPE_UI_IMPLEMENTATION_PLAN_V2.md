# マジカルチェイン 横長UI改造・実装計画 v2

## 1. この文書の目的

- 対象は公開版「カラーチェイン」と、テスト版「クロマのマジカルチェイン」。
- 最初の実装先は `/test/color-chain-mascot/` とし、公開版は検証完了まで現状を維持する。
- 現在の落下、消去、連鎖、特殊ブロック、補助技、得点計算のルールは変更しない。
- 横長UI、スマートフォン操作、キャラクター演出を段階的に導入し、各段階でプレイ可能な状態を保つ。
- 横長UI改造後に、盤面回転式パズルを別URLで試作する方針が追加された。確定仕様と移行手順は `ROTATION_PUZZLE_IMPLEMENTATION_PLAN.md` を参照する。本書の落下式前提は、既存テスト版を維持するための現行仕様として扱う。

## 2. 今回採用する画面方針

ゲーム全体を次の3画面へ分離できる構成にする。

1. マップ画面（将来実装）
   - ドット絵クロマが結び目から結び目へ移動する。
   - ステージ選択、進行状況、浄鎖済みの魔物の再登場を担当する。
2. 会話画面（将来実装）
   - クロマと魔物またはミラの会話を表示する。
   - ステージ固有の背景イラストは主にここで使用する。
3. 対戦画面（今回の優先対象）
   - 華やかな固定背景を全対戦で共用する。
   - 盤面、クロマ、敵、ゲージ、補助技を横長画面に収める。
   - ステージ差は当面、敵キャラクター、UIアクセント、妨害技だけで表現する。

この分離により、全ステージ背景が揃うまで対戦UIの実装を待つ必要がなくなる。後からステージ背景を対戦画面へ追加する場合も、テーマ設定を差し替えるだけで対応できる構造は残す。

## 3. 対戦画面の基本レイアウト

```text
┌────────────────────────────────────────────────────────────┐
│ SCORE  MAX CHAIN  LEVEL                 PAUSE  HELP  SETTINGS │
├──────────────┬──────────────────────┬────────────────────────┤
│ クロマ        │      8 × 16 盤面       │ 敵キャラクター           │
│ 表情・動作     │ CHAIN・技名テロップ     │ 乱れ／封印ゲージ          │
│ 短い台詞       │ 消去・特殊技エフェクト   │ 妨害予告                  │
├──────────────┴──────────────────────┴────────────────────────┤
│ NEXT  HOLD       補助技アイコン＋ゲージ       操作補助ボタン        │
└────────────────────────────────────────────────────────────┘
```

- 盤面を最優先し、キャラクターや演出が操作ブロックを隠さないようにする。
- 対戦背景は中央を暗め、左右を明るめにし、盤面の色視認性とキャラクターの華やかさを両立する。
- PCでは3列、スマートフォン横画面では盤面を中央に固定した圧縮3列を使用する。
- スマートフォン縦画面ではクロマと敵を小型表示または折りたたみ、盤面と操作HUDを優先する。
- ブラウザでは横画面を強制しない。横向き推奨表示は出すが、縦向きでも最後まで操作できる状態を保つ。
- ポータル全体のPWA Manifestを横画面固定にしない。他ゲームまで影響するため、画面方向の案内はマジカルチェイン内部だけで扱う。

## 4. 現状実装から見た前提

- ゲームルールは `src/games/colorChain/logic.ts` に分離済みで、Nodeによるロジックテストがある。
- 画面、入力、進行制御、キャラクター状態は `ColorChain.tsx` に集中している。
- カラーチェイン用CSSは `styles.css` 内にまとまっており、横長UI改造時の影響範囲が大きい。
- 公開版とテスト版は同じコンポーネントを `presentation` で切り替えている。
- 実装用の透過素材は現時点でクロマ5状態のみ。敵キャラクターは設定画段階で、ランタイム用透過素材は未作成。

したがって、最初に表示部品と入力部品を小さく分離し、ゲームルールへ触れずにUIを差し替えられる状態を作る。

## 5. 優先度付き実装計画

### P0-A: 回帰防止とコンポーネント境界の整理

目的: UI改造で既存ゲームルールを壊さない土台を作る。

- `logic.ts` と既存ロジックテストは維持する。
- `ColorChain.tsx` から、盤面表示、ステータスHUD、クロマ枠、補助技HUD、説明・設定画面を順次分離する。
- 状態管理そのものを一度に書き換えず、最初はpropsで既存状態と操作関数を渡す。
- 公開版とテスト版のURL、ベスト記録、ランキングIDを維持する。
- UI変更前の確認項目を手動QAチェックリストとして固定する。

完了条件:

- `npm run test:color-chain` と `npm run build:portal` が成功する。
- 既存のキーボード・画面ボタン操作、NEXT、HOLD、特殊ブロック、補助技、ゲームオーバーが変わらない。

### P0-B: テスト版だけに横長バトルシェルを導入

目的: まず配置と画面密度を確認できる状態にする。

- `/test/color-chain-mascot/` に新しい横長シェルを適用する。
- 上部の成績HUD、左のクロマ枠、中央盤面、右の敵予約枠、下部HUDを配置する。
- 敵ランタイム素材がない間は、右枠を敵未接続の状態表示または非表示可能な予約領域にする。
- 対戦用の固定背景は、初回はCSSの多層グラデーション、光粒子、鎖模様で仮実装する。
- 背景画像を後から追加してもレイアウトを変更しなくてよい構造にする。
- 公開版「カラーチェイン」にはまだ適用しない。

完了条件:

- PC 16:9、タブレット横、スマートフォン横で盤面全体と主要HUDが一画面に収まる。
- 1366×768相当でも縦スクロールなし、または最小限でプレイできる。
- 盤面サイズが小さくなりすぎず、ブロックの色と記号を識別できる。

### P0-C: 縦画面フォールバックと安全領域対応

目的: 横向きにできない端末でもプレイ不能にしない。

- `100dvh`、safe-area、ブラウザUI表示中の高さ変動へ対応する。
- 縦画面ではキャラクター枠を縮小し、盤面直下に操作・補助技を置く。
- 横向き推奨案内は一度閉じられ、ゲーム操作を妨げないものにする。
- `prefers-reduced-motion` と低画質モード用の土台を用意する。

完了条件:

- iPhone相当の狭い縦画面でも盤面、NEXT、主要操作、ポーズへ到達できる。
- 横スクロールを発生させない。

### P1-A: スマートフォンのジェスチャー入力

目的: 盤面から指を大きく離さず直感的に操作できるようにする。

- 短いタップ: 右回転。
- 横スワイプ: セル幅単位で左右移動。
- ゆっくり下スワイプ: 段階的なソフトドロップ。
- 素早い下フリック: ハードドロップ。
- Pointer Eventsでマウス、タッチ、ペンを共通化する。
- `touch-action: none` は盤面だけに指定する。
- `pointercancel`、画面外移動、タブ非表示、ポーズ時に入力状態を必ず解除する。
- キーボードと画面操作ボタンは代替入力として残す。

完了条件:

- タップとスワイプの誤判定が少なく、画面端スワイプと競合しない。
- 入力中にボタンが押しっぱなしになる状態が残らない。

### P1-B: HUD再編と説明・設定の別画面化

目的: 長い説明文を対戦画面から外し、操作領域を近づける。

- NEXT、HOLD、横レーザー、スロータイムを下部のアイコンHUDへ統合する。
- 使用可能、充填中、使用済み、対象選択中を色と形の両方で区別する。
- 説明はPCでドロワー、スマートフォンで全画面シートとして開く。
- 説明を開いたら自動ポーズし、閉じる際の再開確認を安全に扱う。
- 設定項目として、キャラクター表示、画面効果、操作方式、ジェスチャー感度、色覚補助の枠を用意する。
- 日英文言を同時に実装する。

完了条件:

- 主要操作が盤面の近くに集まり、説明を閉じた状態では長文が表示されない。
- すべてのアイコンに名称、ツールチップ、ARIAラベルがある。

### P1-C: ブロック外観とCHAINテロップ

目的: ルールを変えずにゲームの第一印象を刷新する。

- 設定資料の宝石風4色をCSSグラデーションと最小限のSVGで再現する。
- 特殊ブロックも通常ブロックと同じ材質感へ統一する。
- 色だけでなく記号またはルーンを残す。
- 消去地点の近くに `1 CHAIN!`、`2 CHAIN!` などを表示する。
- 特殊技と同時発動した場合は技名を上段、CHAIN数を下段に表示する。
- 表示位置は盤面内へ補正し、操作ブロックを長時間覆わないようにする。

完了条件:

- 通常・特殊ブロック、ゴースト、消去中、NEXT、HOLDで同一デザイン体系になる。
- 低画質・視覚演出軽減設定でも情報が読める。

### P2-A: クロマ演出の横長UI統合

目的: 既存の第1段階マスコット機能を、新しい画面構成へ移す。

- 待機、瞬き、杖回し、帽子、連鎖、危険、敗北、大技反応を左枠で管理する。
- 技名と短い日英台詞をキャラクター枠または一時テロップへ表示する。
- 大技演出は盤面操作を隠さない短時間の背景効果に留める。
- キャラクター表示OFFと視覚演出軽減を維持する。

### P2-B: 敵枠、乱れ／封印ゲージ、妨害予告

目的: 対戦としての基本構造を完成させる。

- 敵ランタイム素材を1体ずつ制作し、最初はモコスライムで接続する。
- 魔物戦は乱れゲージ、ミラ戦は封印ゲージを使用する。
- 軽・中・大被弾と特殊技の対応表を接続する。
- 妨害技は発動前に予告し、何が起きたかを日英で短く表示する。
- 敵が未接続の通常スコアアタックモードも維持する。

完了条件:

- 敵の状態が盤面ルールと同期し、演出をOFFにしても進行状態が理解できる。

### P2-C: 固定対戦背景の最終素材化

目的: 全対戦で使える華やかな共通舞台を完成させる。

- 16:9を基準に、中央盤面用の暗い安全領域と左右キャラクター用の明るい領域を持つ背景を1枚用意する。
- PC・スマートフォン横で同じ画像をトリミングできる構図にする。
- WebPを基本とし、読込中はCSS背景をフォールバックにする。
- 背景と前景粒子を分離し、低画質設定では粒子を停止する。

### P3-A: マップ画面と会話画面

目的: ステージ進行と物語を対戦UIから独立させる。

- ドット絵クロマのマップ移動、結び目選択、ステージ解放状態を実装する。
- 会話画面に通常等身またはデフォルメ立ち絵、敵、日英テキスト、スキップ、ログを配置する。
- ステージ固有背景は会話画面から導入する。
- セーブデータはバージョン付きでlocalStorageへ保存し、将来の追加に耐えられる形にする。

### P3-B: 音声、BGM、ステージ固有演出

- BGM、SE、音量、ミュート、タブ非表示時の停止を実装する。
- 素材名、作者、生成手段、ライセンス、取得元を台帳管理する。
- 必要になった段階で対戦背景のステージ差し替えを検討する。
- カットイン、ボイス、物語演出は基本UIと入力が安定した後に追加する。

## 6. 初回実装スプリントの範囲

最初の実装は次に限定する。

1. UI部品分離の最小限の下準備。
2. テスト版だけに横長3列レイアウトを導入。
3. 固定背景のCSS仮版を導入。
4. 上部ステータスと下部NEXT／HOLD／補助技HUDを再配置。
5. 縦画面のプレイ可能なフォールバックを追加。
6. キーボードと既存画面ボタンの動作を維持。

初回にはジェスチャー、敵ゲージ、新しい宝石ブロック、マップ、会話、BGMを含めない。横長レイアウトの実機確認後、P1を順番に追加する。

## 7. 初回実装前にユーザー側で必要なもの

初回スプリントに必須の追加素材や登録作業はない。既存のクロマ素材とCSS仮背景で開始できる。

後続段階で必要になるもの:

- 固定対戦背景の最終イラスト承認。
- モコスライムから順に、背景透過・同一キャンバスのランタイム用素材を制作する判断。
- BGM／SEの採用素材と利用条件の確認。
- iPhone Safari、Android Chromeの実機操作確認。
- マップ導入前にステージ順、地名、会話量の承認。

## 9. 推奨する着手順の結論

最優先は背景制作ではなく、テスト版の横長シェル、縦画面フォールバック、HUD再配置である。固定背景はCSS仮版で先行し、レイアウトが確定してから最終イラストを作る。ステージ固有背景はマップ・会話画面の実装時に用意する。

ただし、盤面回転式では8×16盤面、NEXT、HOLD、落下操作ボタンが不要になる可能性が高い。これらへ依存するピクセル単位の最終調整は抑え、説明・設定の別画面化、宝石風ブロック、CHAINテロップ、キャラクター・敵・音響など、両方式で再利用できるタスクを優先する。

この順序なら、ゲームとして遊べる状態を保ちながら、横長UI、直感的操作、キャラクター対戦、物語画面へ段階的に発展できる。
